# Baspana Radar — методология

Одностраничник о том, *почему* система устроена так. Это часть
портфолио-артефакта: код показывает «как», этот документ — «почему».

## Задача

Независимый советник покупателя квартиры в Астане. Маркетплейс (krisha.kz)
структурно не может публиковать негатив о своих клиентах-застройщиках;
чат-боты не имеют корпуса данных и уверенно галлюцинируют о конкретных ЖК.
Ниша: слой поверх маркетплейса — расчёты по всему рынку + проверяемые,
градуированные по достоверности факты о конкретных комплексах.

## Данные

Снапшот ~всех объявлений о продаже квартир Астаны с krisha.kz
(search-карточки, не детальные страницы: 20 объявлений/запрос) + каталог
ЖК. PII (имена, телефоны продавцов) вырезается на входе в нормализатор и
не достигает БД. Каждая страница кэшируется до парсинга: повторный парсинг
не трогает сеть. Дедупликация по отпечатку (ЖК, комнаты, площадь, цена ±2%).

## Оценка стоимости

1. **Базлайн**: медиана ₸/м² по ЖК; отклонение объявления от неё.
2. **Hedonic-модель**: gradient boosting на признаках (площадь, комнаты,
   этаж/этажность, год, район, target-encoded ЖК, класс). Модель идёт в
   прод **только если бьёт базлайн на отложенной выборке** — иначе
   скрипт печатает вердикт и остаётся на базлайне. Метрики в
   `valuation_report.txt`.
3. Ограничение честности: предсказываем **asking price**, не цену сделки —
   и говорим это пользователю. ЖК с n<8 объявлений помечаются low_sample,
   их статистика не выдаётся за истину.

## Карточки due diligence

Двухслойные:
- **Layer 1** (все ЖК): LLM видит ТОЛЬКО наш снапшот (параметры ЖК,
  статистику, флаги аномалий, фрагменты объявлений). Разрешённые уровни:
  confirmed (наш расчёт/страница ЖК) и no_data.
- **Layer 2** (флагманские ЖК): вручную собранные источники
  (`pipeline/sources/<slug>/`), LLM структурирует.

Таксономия достоверности: **confirmed** (официальный документ/СМИ) →
**reported** (≥2 независимых источника) → **rumor** (единичное упоминание)
→ **no_data** (обязателен для непокрытых важных тем). Правила enforce'ятся
кодом, не только промптом: флаг без валидного URL бракуется; confirmed без
official/news-источника бракуется; reported с одним источником
автоматически понижается до rumor; личные имена в тексте — брак.

## Аномалии и события

Детекторы: разброс цен внутри ЖК (p75/p25), навес предложения
(доля квартир в продаже), доля «срочно/торг», расхождение заявленного
срока сдачи с фактическими годами постройки (детектор задержек),
устаревшие данные маркетплейса. Инфраструктурный аплифт — курируемая
таблица событий (ЛРТ запущен 16.05.2026 — 18 станций) с
distance-decay-коэффициентами; **коэффициенты — открытые допущения из
литературы о transit premium, не оценки на наших данных**, статус проекта
дисконтируется.

## Оценка качества (eval)

`eval_cards.py`: LLM-судья проверяет каждое утверждение каждой карточки
против исходных данных генератора → доля supported / partial /
unsupported. Число unsupported — наша метрика галлюцинаций; отчёт в
`eval_report.md`. Дизайн-цель: unsupported < 5%, каждое такое утверждение
разбирается вручную.

## Ограничения и путь развития

- Один снапшот = нет временного измерения. С панелью снапшотов (недельный
  re-scrape) открываются: реальные тренды, скорость продажи, и главное —
  **diff-in-diff оценка эффектов инфраструктурных анонсов** вместо
  заимствованных коэффициентов. Это исследовательский апгрейд №1.
- Asking ≠ transaction price; систематический торг ~2-7% не наблюдаем.
- Отзывы 2GIS/Яндекс собираются вручную (анти-бот); покрытие Layer 2
  ограничено кураторским временем — по построению, а не по недосмотру:
  10 честных карточек > 200 галлюцинированных.
